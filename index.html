Project Overview
KloudCampus is a comprehensive SaaS-based college management system designed to streamline operations, enhance institutional compliance, and empower stakeholders with actionable insights.

Project Vision

KloudCampus delivers a unified, enterprise-grade campus management platform that consolidates admissions, academics, examinations, attendance, and regulatory compliance into a single, elegant interface. Built as a multi-tenant SaaS product, it adapts to the unique branding and operational requirements of each institution while maintaining a consistent, professional user experience.

    Role-Based Access Control Architecture
    Granular permission model ensuring secure, role-appropriate feature exposure

    RBAC Framework

    The platform implements a hierarchical RBAC system where permissions are mapped to institutional roles, not individual users. This ensures scalability, maintainability, and consistent policy enforcement across the platform.

    Core Principles

    Principle 1: Principle of Least Privilege

    Each role receives the minimum set of permissions required to fulfill its responsibilities. No user has admin access by default; elevated permissions are explicitly granted and audited.

    Principle 2: Role Hierarchy & Inheritance

    Roles can inherit permissions from parent roles (e.g., Exam Coordinator inherits from Academic Staff). This reduces redundancy and ensures consistent policy cascading.

    Principle 3: Dynamic Feature Visibility

    The UI rendering engine respects role-based permissions in real-time. Navigation menus, buttons, forms, and dashboard widgets are dynamically shown or hidden based on the logged-in user's role and permissions.

    Multi-Tenant Architecture
    Shared application infrastructure with complete data isolation per tenant

    Architecture Philosophy

    KloudCampus follows a **shared-application, isolated-tenant** architecture model. This approach combines the operational efficiency of SaaS delivery with the security and compliance requirements of institutional deployments. Each college operates as an independent tenant within the unified platform while benefiting from continuous feature updates, security patches, and infrastructure improvements delivered across the entire system.

    Core Principle: Shared Infrastructure, Isolated Data

    One application codebase powers all institutions. Each tenant's data remains completely isolated at the database, file storage, and API levels. Institutions never share databases, files, or resources with other institutions. Updates to the platform benefit all tenants simultaneously without requiring individual deployment or migration efforts.

    Key Architecture Components

    1. Shared Application Layer

    A single, unified codebase serves all tenants. The application is stateless and horizontally scalable across multiple server instances. All instances run identical code, ensuring consistent behavior across the platform.

    • Containerized Deployment: Docker containers ensure consistency across environments. Kubernetes orchestration manages scaling and failover.
    • Load Balancing: Multi-region load balancers distribute traffic based on geographical location and system load, ensuring low latency.
    • Auto-Scaling: Horizontal pod autoscaling responds to demand spikes, maintaining performance during peak usage periods.
    • Unified Code Deployment: Feature updates, security patches, and performance improvements deploy once to benefit all institutions simultaneously.

    2. Tenant Identification & Routing

    Every request includes a tenant identifier that routes to the correct data isolation boundary. Tenant identification occurs at multiple layers for security.

    • Domain-Based Routing: Each institution has a custom subdomain (e.g., stanford.kloudcampus.io, mit.kloudcampus.io) or white-labeled domain.
    • JWT Token Encoding: User authentication tokens include the tenant ID, preventing cross-tenant access.
    • Database Connection Pooling: Tenant-aware connection pools maintain isolated database connections per tenant.
    • API Gateway Layer: Validates tenant context before routing requests to backend services.

    3. Isolated Data Storage

    Each tenant's data is completely isolated at the database and file system levels. No queries can accidentally access another institution's data.

    Data Component Isolation Strategy Implementation
    Relational Data Database per Tenant OR Schema per Tenant PostgreSQL databases or schemas with row-level security policies
    File Storage S3 bucket prefixes or separate buckets per tenant AWS S3 with IAM policies restricting access to tenant-specific prefixes
    Cache Layer Namespaced Redis keys per tenant Redis with tenant_id prefixes (e.g., tenant:12345:user:john)
    Search Indexes Elasticsearch indices per tenant Separate Elasticsearch indices with tenant routing
    Message Queues Tenant-prefixed topics in message brokers RabbitMQ or Kafka with tenant-scoped channels
    Logs & Audit Trails Tenant-aware logging with filtering ELK Stack with Logstash filtering by tenant_id

    5. Mandatory Columns on Every Table

    Every table in KloudCampus must include these columns without exception. This is enforced through code review checklists and database migration linting. No pull request is merged without these columns present.

    Column Type Constraint Purpose
    tenant_id UUID NOT NULL, FK → tenants(id) Isolates every row to a specific college. Without this column, data leaks across tenants.
    created_by UUID FK → users(id) Tracks which user created the record. Essential for audit trail and accountability.
    created_at TIMESTAMPTZ NOT NULL, DEFAULT now() Exact timestamp when the record was created (UTC). Used for audit, reporting, and compliance.
    updated_at TIMESTAMPTZ NOT NULL, DEFAULT now() Last modification timestamp. Auto-updated via trigger on every UPDATE operation.
    Design Rule: Tenant-Scoped Uniqueness

    All UNIQUE constraints must include tenant_id as a composite key. For example, enrollment number "CS2024001" can exist independently in College A and College B. Without tenant-scoped uniqueness, the second college to use that enrollment number would receive a database constraint violation.

    6. Database Isolation Strategy Options

    KloudCampus supports three isolation strategies, selected based on the institution's subscription plan and data sovereignty requirements.

    Strategy How It Works Best For Plan
    Shared Database + Row Isolation (Recommended) Single database, tenant-scoped queries enforced at ORM level and PostgreSQL Row-Level Security. Every query is filtered by tenant_id automatically. 95% of institutions. Best balance of cost, simplicity, and security. Starter & Growth
    Shared Database + Schema Isolation Single database server with a separate PostgreSQL schema per tenant. Each schema contains identical table structures but completely separate data. Institutions requiring stronger logical separation without dedicated infrastructure cost. Growth & Enterprise
    Separate Database per Tenant Dedicated database instance per institution. Full physical isolation. Independent backup, restore, and scaling per tenant. Enterprise contracts requiring data sovereignty, regulatory compliance, or on-premise deployment options. Enterprise Only

    7. Data Isolation Implementation Details

    Database Isolation Strategy: Row-Level Security (RLS)

    KloudCampus uses PostgreSQL Row-Level Security policies to enforce tenant isolation at the database level. Every table has a tenant_id column, and RLS policies prevent unauthorized access.

    
-- Enable RLS on all tables
ALTER TABLE students ENABLE ROW LEVEL SECURITY;
ALTER TABLE courses ENABLE ROW LEVEL SECURITY;
ALTER TABLE grades ENABLE ROW LEVEL SECURITY;

-- Create policy: Users can only see data from their tenant
CREATE POLICY tenant_isolation ON students
  USING (tenant_id = current_setting('app.current_tenant_id')::uuid);

CREATE POLICY tenant_isolation ON courses
  USING (tenant_id = current_setting('app.current_tenant_id')::uuid);

-- Set tenant context for each connection
SET app.current_tenant_id = '12345-67890-abcde-fghij';

-- Query returns only tenant-specific rows
SELECT * FROM students; -- Only returns students from tenant 12345

    6. Scaling Architecture

    The isolated-tenant model allows sophisticated scaling strategies:

    • Horizontal Pod Autoscaling: Application pods scale based on CPU/memory utilization, ensuring responsive performance.
    • Database Read Replicas: High-read institutions benefit from read replica configurations without affecting others.
    • Cache Warming: Tenant-specific cache pre-loading optimizes performance for large institutions.
    • Queue Isolation: Background jobs for different tenants run on isolated worker queues, preventing one institution from blocking others.
    • CDN Distribution: Static assets and media files distributed via CDN, cached per tenant namespace.

    Security & Compliance Measures

    Data Isolation Security

    • Network Isolation: Virtual Private Cloud (VPC) per environment ensures network-level isolation.
    • Database Encryption: All databases encrypted at rest (AES-256) and in transit (TLS 1.3).
    • Key Management: Encryption keys per tenant stored in AWS KMS with access policies.
    • Field-Level Encryption: Sensitive data (SSN, passwords, financial records) encrypted at application level.
    • Audit Logging: All data access logged per tenant with immutable audit trails.
    • Cross-Tenant Query Prevention: Database triggers prevent JOIN queries across tenant boundaries.

    Tenant Suspension & Data Retention

    • Graceful Suspension: When a subscription ends, tenant access is suspended immediately. Data retained per contract terms.
    • Data Deletion Workflow: On institution request, data deletion follows legal hold and compliance requirements.
    • Backups Compliance: Institution data is backed up independently, retained per SLA terms.
    • Right to Portability: Institutions can request full data export in standard formats (CSV, JSON, SQL dumps).

    Performance & Optimization

    Multi-Tenant Performance Considerations

    • Noisy Neighbor Problem: Prevent large institutions from monopolizing resources. Use query limits, connection pool quotas, and rate limiting per tenant.
    • Index Optimization: Composite indexes on (tenant_id, entity_id) for fast tenant-scoped lookups.
    • Connection Pooling: PgBouncer connection pooling with per-tenant quotas prevents connection exhaustion.
    • Query Caching: Redis caching with tenant namespace reduces database load for frequently accessed data.
    • Batch Processing: Background job isolation ensures one tenant's large export doesn't block another's data import.

    Monitoring & Observability

    • Per-Tenant Metrics: Prometheus metrics labeled by tenant_id for granular monitoring.
    • Distributed Tracing: Jaeger traces requests across services, including tenant context for debugging.
    • Resource Quotas: Alert when tenants exceed CPU, memory, or database quota thresholds.
    • SLA Dashboards: Grafana dashboards showing uptime, latency, error rates per tenant.

    Billing Features

    The billing engine handles the complete subscription lifecycle from trial signup to renewal, including payment collection, invoice generation, plan changes, and account suspension for non-payment.

    Subscription Management

    • Trial Period Logic: 14-day free trial on Growth plan with full feature access. No credit card required for trial signup. Automatic downgrade to Starter if no payment method added before trial expiry. Trial extension available via sales approval.
    • Plan Upgrades: Instant upgrade with prorated billing. If an institution upgrades from Starter to Growth mid-cycle, they are charged the difference for the remaining days. Feature access is unlocked immediately upon payment confirmation.
    • Plan Downgrades: Effective at end of current billing cycle. Data is retained but features beyond the new plan become read-only. If student count exceeds new plan limit, institution must reduce active students before downgrade takes effect.
    • Annual vs Monthly: Monthly billing with automatic renewal. Annual billing at 20% discount with upfront payment. Annual contracts include a 30-day cancellation window after renewal.

    Payment Processing

    • Payment Gateways: Razorpay (primary for India — UPI, net banking, cards, wallets), Stripe (international payments — cards, ACH, SEPA). Gateway selection is automatic based on institution's billing country.
    • Invoice Generation: Automated invoice generation on each billing cycle. GST-compliant invoices with GSTIN, HSN/SAC codes, and proper tax breakdowns. Invoices available for download in PDF format from the admin dashboard.
    • Tax Calculation: India: GST at 18% on SaaS services (IGST for inter-state, CGST+SGST for intra-state). International: VAT handling based on institution's country. Tax rates configurable per region and updated quarterly.
    • Payment Tracking: Real-time payment status dashboard showing pending, completed, and failed transactions. Automatic receipt generation and email delivery upon successful payment. Payment history accessible to institution admins.

    Dunning Management (Failed Payment Recovery)

    • Retry Schedule: Day 0: Payment fails → automatic retry. Day 3: Second retry + email notification to billing admin. Day 7: Third retry + SMS/WhatsApp alert to institution principal. Day 14: Account enters "grace period" (read-only access). Day 30: Account suspended (no access, data retained for 90 days).
    • Recovery Rate: Industry benchmark: 70% of failed charges succeed on first retry. KloudCampus targets 85% recovery through multi-channel reminders and flexible payment options.
    • Reactivation: Suspended accounts can be reactivated by clearing outstanding balance. All data is intact. Feature access restored within minutes of payment confirmation.

    Usage Analytics & Metering

    • Student Count Metering: Active students counted as students with at least one login or attendance record in the current billing period. Historical/alumni students do not count toward the cap. Real-time counter visible in admin dashboard.
    • Storage Metering: File uploads (documents, photos, transcripts) tracked per tenant. Starter: 50GB, Growth: 200GB, Enterprise: custom. Overage alerts at 80% and 95% capacity.
    • API Rate Limiting: Starter: 500 requests/minute, Growth: 2,000 requests/minute, Enterprise: custom. Rate limit headers included in every API response. Burst allowance of 2x for 30-second windows.

    Workflow Architecture

    KloudCampus replaces manual paper-based processes with digital workflow automation. Each workflow is a state machine with defined transitions, approval gates, and notification triggers. Workflows ensure that no critical process is skipped, every approval is recorded, and every stakeholder is notified at the right time.

    Admission Workflow

    The admission workflow handles the complete lifecycle from online application to student enrollment. Each state transition requires explicit action from an authorized role, creating a full audit trail.

    Step State Action Responsible Role Notification Trigger
    1 Application Submitted Applicant fills online form, uploads documents, pays application fee Applicant (self-service) Email + SMS confirmation to applicant
    2 Document Verification Admin reviews uploaded documents for authenticity, completeness, and eligibility Admissions Officer In-app task notification to Admissions Officer
    3 Merit Evaluation System calculates merit score based on academic records, entrance exam, category reservation System (automated) + Registrar review None (internal processing)
    4 Principal Approval Final authority reviews merit list and approves selected candidates Principal / Director In-app notification to Principal with approval queue
    5 Offer Generation System auto-generates admission offer letter with enrollment details, fee structure, reporting date System (automated) Email + WhatsApp offer letter to applicant
    6 Fee Payment & Enrollment Applicant pays admission fee. Upon confirmation, student account is auto-created with enrollment number. Applicant + System Welcome email + SMS with login credentials

    Exam Result Workflow

    The exam result workflow ensures grades go through proper verification and approval before students can see them. This prevents unauthorized grade modifications and creates an auditable approval chain.

    Step State Action Responsible Role Notification Trigger
    1 Marks Entry Faculty enters marks for each student in their assigned courses. System validates marks against max_marks. Faculty / Instructor Reminder notification if marks not entered 48 hours after exam date
    2 Faculty Verification Faculty self-reviews entered marks, checks for errors, and locks the submission. No edits allowed after lock. Faculty / Instructor Confirmation to faculty upon successful lock
    3 HOD Approval Department head reviews marks distribution, checks for anomalies (unusually high fail rate, suspicious patterns) Head of Department In-app notification to HOD when faculty locks submission
    4 Registrar Approval University-level compliance check. Registrar verifies grade distribution, applies grace marks if applicable, approves for publication. Registrar In-app notification to Registrar after HOD approval
    5 Result Publication Approved results are published to student dashboards. Transcripts become downloadable. Grade cards generated. System (automated on Registrar approval) Email + WhatsApp + Push notification to all affected students

    Leave Request Workflow

    Step State Responsible Role
    1Leave Request Submitted — Faculty/staff submits leave request with dates, type (casual/medical/earned), and reasonFaculty / Staff
    2HOD Review — Department head reviews and approves/rejects based on workload impact and substitute availabilityHOD
    3Principal Approval — Required only for leaves exceeding 3 days or special leave typesPrincipal
    4HR Processing — Leave balance updated, substitute assigned if applicable, payroll adjusted if unpaid leaveSystem + HR

    Workflow Engine Configuration

    • State Machine Design: Each workflow is a finite state machine with defined states, transitions, and guards. States are stored in the database with the current state of each entity (application, grade submission, leave request). Transitions require both a valid action and an authorized role.
    • Conditional Branching: Workflows support conditional logic. For example, leave requests under 3 days skip Principal approval and go directly to HR processing. Merit evaluation automatically routes to a different approval chain based on reservation category.
    • Timeout and Escalation: If an approval step is not actioned within the configured SLA (e.g., HOD does not approve grades within 48 hours), the system sends a reminder. After a second timeout, the request escalates to the next authority level (Registrar or Principal).
    • Notification Integration: Every state transition triggers a configurable notification. The notification channel (in-app, email, SMS, WhatsApp) and template are defined per transition per workflow. Institutions can customize notification templates.
    • Audit Trail Integration: Every workflow transition is recorded in the audit_logs table with the old state, new state, action taken, user who performed the action, timestamp, and IP address. This provides irrefutable evidence for compliance and dispute resolution.
    • Parallel Approval: Enterprise plan supports parallel approval flows where multiple approvers must sign off before proceeding. For example, both HOD and Exam Coordinator must approve a revaluation request before the Registrar can process it.

    Audit Trail
    Immutable activity logging for compliance, accountability, and dispute resolution

    Why Audit Trail is Critical

    In a college management system, data integrity is paramount. Grades determine student futures. Attendance records affect eligibility. Fee records have financial and legal implications. The audit trail provides irrefutable evidence of who changed what, when, from where, and what the data looked like before and after the change. This is not optional — it is a legal and compliance requirement for NAAC, AICTE, and institutional governance standards.

    What Must Be Tracked

    • Grade Modifications: Who changed a student's marks? What was the original score? What is the new score? When was it changed? Was it approved by HOD?
    • Attendance Changes: Who marked attendance? Was it retroactively modified? Original status vs updated status. IP address of the device used.
    • Admission Decisions: Who approved the admission? Who rejected it? What was the merit score at the time of decision?
    • Transcript Generation: Who generated the transcript? When? How many times? Was it an official or duplicate copy?
    • Fee Structure Changes: Who modified the fee structure? What was the old amount? What is the new amount? Was it applied retroactively?
    • Record Deletions: Who deleted a record? What was the complete data before deletion? Soft-delete is enforced; hard-delete requires Super Admin + audit justification.

    Audit Log Schema

    Column Type Purpose
    idBIGSERIAL PKAuto-increment primary key
    tenant_idUUID NOT NULLWhich college (tenant isolation applies to audit logs too)
    user_idUUID NOT NULL (FK → users)Who performed the action
    actionVARCHAR(10) NOT NULLCREATE, UPDATE, DELETE, VIEW, EXPORT, LOGIN, LOGOUT
    entityVARCHAR(100) NOT NULLTable or module affected (e.g., "grades", "attendance", "students")
    entity_idUUID NOT NULLPrimary key of the affected record
    old_valueJSONBComplete previous state of the record (null for CREATE actions)
    new_valueJSONBComplete new state after the change (null for DELETE actions)
    ip_addressINETClient IP address for security forensics
    user_agentTEXTBrowser/device information for device identification
    created_atTIMESTAMPTZ NOT NULLExact timestamp of the action (UTC, millisecond precision)

    Audit Trail Rules

    • Immutability: Audit logs cannot be updated or deleted by any user, including Super Admins. Database rules (PostgreSQL RULE) prevent UPDATE and DELETE operations on the audit_logs table. Only INSERT is allowed.
    • Tenant Isolation: Row-Level Security applies to audit logs. College A's admin cannot view College B's audit trail. Platform admins (KloudCampus internal) have cross-tenant read access for support and investigation.
    • Retention Policy: Audit logs retained for minimum 7 years (aligned with Indian regulatory requirements). Archived to cold storage (S3 Glacier) after 2 years. Always retrievable within 48 hours for compliance audits.
    • Search and Filter: Audit logs are searchable by entity type, user, date range, action type, and entity_id. Full-text search on old_value and new_value JSONB fields for investigating specific data changes.
    • Compliance Reporting: One-click audit report generation for NAAC, AICTE, and institutional governance reviews. Reports show all data modifications within a specified date range, grouped by module and user.

    Multi-Tenant Customization Architecture
    Tiered branding and theming system allowing institutional personalization

    Multi-Tenant Model Overview

    KloudCampus is architected as a true SaaS platform with isolated data tenants but shared application code. This approach ensures:

    • Rapid updates and feature rollouts across all institutions
    • Each college maintains complete data isolation and privacy
    • Flexible customization options without fragmenting the codebase
    • Consistent core user experience with room for institutional identity

    Customization Tiers

    KloudCampus offers three tiers of customization, allowing institutions to choose the level of personalization appropriate to their needs and technical capability.

    Tier 1
    Essential Branding (Standard Plan)

    Pre-configured templates with basic institutional customization. Ideal for colleges seeking simplicity and rapid deployment.

    • Upload institution logo (SVG or PNG)
    • Choose from predefined color theme sets
    • Customize primary institution name and abbreviation
    • Upload favicon for browser tab
    • Modify footer text and links
    • Email signature branding
    • Standard pre-configured landing page

    Deployment: Configuration through web-based admin panel. No code required. Changes deploy within minutes.

    Tier 2
    Advanced Theming (Professional Plan)

    Granular control over colors, typography, and layout. Suitable for institutions with specific brand guidelines and technical resources.

    • Full color palette customization (primary, secondary, accent, semantic colors)
    • Custom Google Font integration
    • Adjust spacing and border radius system-wide
    • Custom CSS variable overrides via admin dashboard
    • Create custom role-specific color schemes
    • Department-level sub-branding within single tenant
    • Custom email templates
    • Logo placement and sizing configuration

    Deployment: WYSIWYG theme editor in admin console. Preview changes in real-time before publishing.

    Tier 3
    Custom Development (Enterprise Plan)

    Full code-level customization for organizations requiring bespoke functionality or deeply integrated branding.

    • Custom CSS stylesheet injection
    • Branded React component overrides
    • Custom API integrations (SIS, ERP, LMS)
    • White-label deployment options
    • Custom domain and SSL certificate
    • Dedicated support and custom deployment
    • Custom business logic and workflows
    • On-premise deployment option

    Deployment: Managed by KloudCampus professional services team with dedicated DevOps support.

    Deployment & Theme Updates

    Theme updates are applied through an admin-controlled publish mechanism, ensuring institutional control over when changes become visible to end-users.

    Theme Update Workflow

    1. Draft: Admin makes customization changes in the theme editor (color picker, font selector, etc.)
    2. Preview: Live preview shows changes in a side-by-side comparison with production
    3. Test: Optional: Deploy to staging environment for broader user testing
    4. Publish: Admin approves and publishes theme. Changes deploy via CDN with <1 second propagation
    5. Rollback: One-click rollback to previous version if issues arise
    Theme Versioning & Audit

    All theme changes are versioned and logged. Audit trail tracks who made changes, when, and what was modified. This supports compliance requirements and institutional governance.

    Multi-Agent Development Framework
    Claude-powered agent orchestration for collaborative software development and maintenance

    Framework Overview

    KloudCampus development leverages a collaborative multi-agent system where specialized Claude agents assume distinct roles in the software development lifecycle. Each agent has focused expertise, clear responsibilities, and defined communication protocols. This approach enables parallel development workflows, automated testing, quality assurance, and continuous deployment — all coordinated through intelligent agent orchestration.

    Agent-Driven Development Philosophy

    Rather than traditional siloed teams (frontend team, QA team, DevOps team), the multi-agent framework enables agents to collaborate dynamically on tasks, review each other's work, identify issues proactively, and self-organize around priority and dependencies. Each agent has access to the codebase, requirements, and real-time feedback from other agents.

    Example: Multi-Agent Development of "Admissions Module"

    Step-by-Step Collaboration

    Day 1 — Design & Architecture

    System Architect Agent analyzes admissions requirements, designs data models, API endpoints, and creates architecture document. Architecture includes entity relationships (Applicant → Application → Merit List → Enrollment), API endpoints (/api/admissions/applications, /api/admissions/offers), and authentication flows.

    Day 2 — Parallel Development

    Three agents work in parallel:

    • Backend Developer Agent: Implements Express.js APIs, database operations, and business logic. Creates endpoints for application submission, merit list generation, offer generation.
    • Frontend Developer Agent: Builds React components for application form, applicant dashboard, admin controls. Implements form validation and file upload for documents.
    • Database Specialist Agent: Optimizes schema, creates indexes on applicant status and merit scores, designs migration scripts.

    Day 3 — Testing & Quality

    Three agents focus on quality:

    • QA Testing Agent: Writes unit tests for API logic, integration tests for database operations, end-to-end tests for user workflows. Tests edge cases like duplicate applications, missing documents, merit score calculations.
    • Security Agent: Reviews API authentication, checks for SQL injection risks, validates file upload security, ensures RBAC (only Registrar can generate offers).
    • Code Review Agent: Analyzes code quality, checks adherence to project standards, identifies performance issues, verifies test coverage.

    Day 4 — Integration & Deployment

    • Integration Agent: Tests API contracts match expectations, verifies frontend-backend integration, tests multi-tenant data isolation.
    • DevOps Agent: Builds Docker image, runs CI/CD pipeline, deploys to staging, executes smoke tests, deploys to production.
    • Documentation Agent: Generates API documentation, creates user guides for admissions officers, documents deployment steps, generates release notes.

    Day 5 — Monitoring & Iteration

    • Monitor production deployment for errors
    • Gather feedback and log issues
    • Iterate on discovered bugs or improvements
    • Update documentation based on real-world usage
